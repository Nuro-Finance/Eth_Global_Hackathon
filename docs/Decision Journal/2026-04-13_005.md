# Debug LayerZero 2-Hop Bridge — zkSync → Arbitrum

## Context
Testing the exotic 2-hop bridge: zkSync → LayerZero → Arbitrum → CCTP → Base → Issuer card.

## Root Causes Found (3 bugs)

### Bug 1: Options Length Encoding (THE BIG ONE)
Options V3 format: `version(2) + workerID(1) + optionLength(2) + type(1) + gas(16) + value(16)`
Option data = type(1) + gas(16) + value(16) = **33 bytes = 0x0021**
We had: **0x0011 (17 bytes)** — parser read 17 bytes of option data, cursor landed in the middle of the value field, interpreted `0x00` as the next worker ID → `InvalidWorkerId(0)`

### Bug 2: Wrong DVN Address Per Chain
Our layerzero.config.ts had DVN `0x620A9DF7...1A4dE1b5` for zkSync.
Correct DVN from LZ metadata API: `0x620a9df7...013f5c51`.
First 18 bytes matched, last bytes diverged. DVN addresses are DIFFERENT per chain.

### Bug 3: Arb Adapter Had 0 USDC
OFTAdapter uses lock/release pattern. Nobody had ever locked USDC into the Arb adapter, so `_credit` reverted on delivery. Fixed by seeding adapter with 0.20 USDC.

## Result
- LZ send tx confirmed on zkSync: `0xb01c2e4f...`
- DVN verification confirmed on Arb: `0x1f78c055...`
- Executor delivery confirmed: `0xccc8f199...`
- Adapter released 0.09025 USDC to deployer wallet
- Status: DELIVERED

## Lessons Learned
1. Options encoding must be EXACT — one byte wrong = hours of debugging
2. DVN addresses differ per chain — always verify from metadata API
3. OFTAdapter needs USDC seeded on the hub chain before first use
4. Richard was right — LayerZero DOES support zkSync, the infrastructure IS there
