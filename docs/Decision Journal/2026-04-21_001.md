## DJ 1 — Gate-check blocked the webhook flip, and that was the right outcome

**Question**: Session 27 opened with Owen green-lit webhook delivery resumption. Natural instinct: flip `ISSUER_WEBHOOK_OBSERVE_ONLY=false` and ship. The new `issuer-webhook-live` gate I wrote last session would auto-run if I invoked `/gate-check`.

**Options considered**:
- (A) Trust the handoff — flip the env, pm2 restart, call it done.
- (B) Actually run `/gate-check issuer-webhook-live` as promised.

**Chose**: (B). And it FAILED CLOSED — `ISSUER_WEBHOOK_SECRET` was missing from `.env` on VPS. If I'd flipped observe-only to false without the secret set, the webhook verifier would no-op on every incoming request → SD3 events bypass HMAC → arbitrary POST could update `card_transactions` + `cards.balance` + `users.kyc_status`.

**Why**: gate-check is only useful if we actually use it. Writing a gate and then skipping it is worse than no gate — it creates false confidence.

**Tradeoff**: 15 min of "delay" finding the missing secret. But a real incident if skipped — unauth'd writes to `users.kyc_status` would open an auth bypass.

**Verdict at session close**: The gate caught the exact class of issue it was designed for. Every future `/gate-check` invocation reinforces the habit. Doing the thing you said you'd do compounds.

## DJ 2 — SD3 "outage" was a misdiagnosis, walking it back publicly matters

**Question**: Richard's alarm-level question — "if SD3 stays dark we have no platform" — landed after I'd stated flatly "webhook delivery stopped Apr 17, SD3 might be down." Under that framing, the sensible path is "start migrating to a second BIN sponsor."

**Options considered**:

**Chose**: (B). Curl returned HTTP 200 in 214ms. API was never down — only the webhook delivery side stopped. I had misread the execution_log (100% 429s for 24h) as "SD3 throttling" when actually balance-sync was logging-skipped-on-drift<$0.01 for days (no balance change = silent skip).

**Why**: Richard spent 3 minutes genuinely scared we'd lost the card path. Staying with the wrong framing would have cost an hour of unnecessary "evaluate BIN alternatives" planning. Walking it back in the same session meant we refocused on the actual ask (the single narrow Owen question: re-enable webhook config + paste secret).

**Tradeoff**: Admitting I was wrong in a conversational reply. That's a trivial tradeoff against wasting an hour on a phantom crisis.

**Verdict at session close**: Correct call. Owen responded in 10 min and the path forward is clear. The cost of being wrong once is much smaller than the cost of sticking with wrong.

## DJ 4 — User-ID migration: migrate data AND code atomically, keep HD derivation seed unchanged

**Question**: `transactions.user_id` and `deposit_addresses.user_id` had been storing SD3 UUIDs instead of local `users.id`. Migration was clear (Session 26 tech-debt note), but execution had three subtle coupling risks.

**Options considered**:
- (A) Migrate data first, deploy code later. Simple sequencing.
- (B) Deploy code first (query via local userId), then migrate data. Reads return empty during the window.
- (C) Migrate data AND code in the same `git push` + `pm2 restart`. Atomic.

**Chose**: (C). Plus a non-obvious call: **keep `generateDepositAddress(sd3UserId)` HD-seed path unchanged**. Only the DB column value migrates. On-chain addresses must stay stable so existing users' past deposits are still reachable.

**Why**: (A) would leave line 782's `WHERE user_id = $issuer_user_id` query returning rows that no longer exist at that column value — zero-row responses for minutes. (B) symmetric problem. (C) makes the window effectively zero. Separately: if I'd also changed the HD seed, new invocations would generate NEW addresses that don't match what's recorded on-chain — every pre-migration deposit becomes orphaned.

**Tradeoff**: (C) requires careful commit/push/pull/restart sequencing. One atomic deploy. Did it, verified with a JWT-signed curl against `/card-transactions` — $6.84 total income (same as before migration). Zero behavioral drift.

**Verdict at session close**: Clean migration. 73 transactions + 3 deposit_addresses updated. Orphans (test fixtures + one failed SD3 session) left alone with a written rationale. Rollback table captured for paranoia.

## DJ 5 — TypeScript config upgrade: not a rewrite, just honest targets

**Question**: We'd been ignoring a ~100+ error TypeScript noise for weeks — BigInt literals, Set/Map iteration, function-in-blocks. Options: chase each error, or fix the config.

**Options considered**:
- (A) Leave it. Next.js has `ignoreBuildErrors: true` so nothing actually breaks. Backend uses `tsx` so runtime is unaffected.
- (B) Bump `target: es5 → es2020` and `moduleResolution: node → bundler`. Exclude cashly-vaults + dead scripts.

**Chose**: (B). Error count went 110 → 38, a 65% reduction, in one commit. Zero runtime change.

**Why**: The noise was actively hiding real errors. Every new PR diff would show 100 existing errors + 1-2 new ones, and the new ones got lost. With 38 errors we can actually see when something new regresses. Also: ES5 target in 2026 is just wrong — the runtime (Node 18+/Next.js 15) is way past that.

**Tradeoff**: 38 remaining errors are real type debt we should chase eventually. But they're real bugs, not noise — much more actionable at this level.

**Verdict at session close**: Dev velocity win. `tsc` output stops being a scroll-past. Future sessions can actually use tsc as a quality gate.

## DJ 6 — MCP SSH: did NOT install, wrote a decision log instead

**Question**: Marathon 3 Sprint 3.1 item "MCP SSH" was on the todo. Researched the landscape (ssh-mcp-server, SSH-MCP, mcp-server-ssh) + compared to existing Bash tool.

**Options considered**:
- (A) Install `@fangjunjie/ssh-mcp-server` with command-allowlist regex. ~15 min setup.
- (B) Leave Bash as-is, document the trade-off + when to revisit.

**Chose**: (B). Wrote `MCP Integrations Decision Log.md` documenting every MCP we've evaluated, what's installed, what's deferred, and a rubric for future additions.

**Why**: The primary value of MCP SSH over Bash is command-whitelist enforcement — useful for multi-operator setups. We're single-founder. Every SSH call so far has been Bash-based with zero friction. Installing for sake of installing is the opposite of engineering.

**Tradeoff**: If we hire operators later, we'll revisit. If the Bash shell-escaping complexity ever bites (long heredocs, nested quotes) we'll revisit. Neither is true today.

**Verdict at session close**: The decision log is the output. Future-me will read the rubric ("default answer: don't install, the bar should be high") and not re-scout this from scratch.

## DJ 7 — Solana Privy: defensive require + dynamic import so Session 20 doesn't repeat

**Question**: Session 20 Privy outage (full-site crash from a placeholder env var) is still fresh. Adding Solana support meant importing `@privy-io/react-auth/solana` submodule which may not exist on older Privy versions.

**Options considered**:
- (A) `import { useSolanaWallets } from '@privy-io/react-auth/solana'` at the top — fails loudly if missing.
- (B) Runtime `require()` inside a try/catch — silently falls back to null address if the submodule isn't there.

**Chose**: (B). Plus a `ready` state + `hasSolanaWallet` bool so consumers can render empty-state instead of crashing.

**Why**: Privy Solana support is v2.x only. If someone downgrades Privy (not planned, but possible), the static import breaks the whole FE build. Dynamic require isolates the risk. Same defensive pattern we used for `usePrivyWalletAddress`.

**Tradeoff**: Slightly uglier code + no tree-shake benefit (can't dead-code-eliminate a dynamic require). Acceptable cost for Session-20-outage prevention.

**Verdict at session close**: Code shipped. No UI wired yet — the hook is dormant, waiting for `/my-wallet` Solana tab work. But the data layer + Privy config + address-resolver are all in place. Session 28 work can just consume the hook.
