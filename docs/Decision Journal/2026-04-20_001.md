# Decision Journal — Session 26 (2026-04-20)

Session 26 spanned wallet multi-chain polish, a real-data rewrite of the Overview allocation chart, Sprint 2.4 (Visa spend) polish, Sprint 2.1 vault exposure, Sprint 6.5 observability (three sub-pieces), and a "make the growth agent smarter" cron. This DJ captures the non-obvious calls behind those.

---

## DJ 1 — Multi-chain Sell: `switchChainAsync` on quote vs block until user switches manually

**Question**: When a user opens the Sell tab on Polygon and picks a Base-native token, how does the firm-quote fetch know to use the right chain ID?

**Options considered**:
- (A) Block the quote request until the user manually switches network via MetaMask. Show a "Switch to Base to continue" banner. No accidental switches.
- (B) Call `switchChainAsync(token.chainId)` from wagmi right before the firm-quote POST. Fewer clicks, chain is right when 0x signs.

**Chose**: (B) — auto-switch on quote.

**Why**: The whole point of "multi-chain sell" is the user doesn't have to think about which chain their USDC is on. Forcing a manual switch per token defeats the premise. `switchChainAsync` throws if the user rejects, which we catch and surface as an inline error — so the consent model is preserved.

**Tradeoff**: One extra network permission prompt per sell attempt on a non-active chain. Mitigated because wallets remember the approval after the first switch per session.

**Verdict at session close**: Works cleanly. The CTA label flips to "Switch to Base & Sell" when the chain mismatches, so there's no surprise.

## DJ 2 — Token whitelist: positive allowlist vs tweak the scam-scoring thresholds

**Question**: The scam filter was stripping legit tokens like LINK and PEPE at $0 price (pricefeed stale). Two ways to fix.

**Options considered**:
- (A) Lower the "suspicious if price=$0" weight in `looksLikeScam()`. Effectively make the heuristic more forgiving.
- (B) Add a positive whitelist — hook fetches `/api/supported-tokens`, builds Sets of addresses + symbols, and `looksLikeScam()` early-returns `false` on match.

**Chose**: (B) — positive allowlist overrides everything else.

**Why**: Relaxing heuristics lets in real scams. We've already curated the supported-token list server-side (audited, known-safe). If a token is on that list, it's definitively legit — no heuristic can override that ground truth. The allowlist is additive: we keep the strict heuristic for unknown tokens and trust the curated list for the long tail of legit ones with stale pricing.

**Tradeoff**: Two sources of truth (whitelist + heuristic). Mitigated because they're applied in order (whitelist wins), and the whitelist is module-level cached with 10-min TTL so we're not fetching every render.

**Verdict at session close**: Clean. LINK/UNI/SHIB/PEPE now render correctly even when CoinGecko is stale.

## DJ 3 — Sprint 2.4 polish: merchant name from `merchant_name` column vs force schema migration

**Question**: The admin already had `issuer-sync.ts` wired to pull SD3 webhook events with merchant metadata. The missing piece was the FE showing that metadata. Where to put it?

**Options considered**:
- (A) Add merchant_name + MCC to the `transactions` FE type by extending `rowToTransaction()` in `nuro-routes.ts`, branching the description cell on `merchantName` presence.
- (B) New table `card_spend_metadata` joined at query time. Clean separation of "bridge txs" from "Visa txs".

**Chose**: (A) — extend the row-mapper, branch in the UI.

**Why**: The data is *already* on the same `transactions` row (issuer-sync writes merchant_name + merchant_category_raw directly). Splitting into a joined table is architecturally purer but adds one migration, one extra query, and a nullable-join on every list load. Branching in the UI costs 12 lines of JSX. The purer model can wait until we have >1 non-bridge tx type (gift cards? payouts? then split).

**Tradeoff**: `Transaction` interface now carries 5 optional Visa-specific fields that are null for bridge txs. Every consumer that adds a new column type has to know to handle null.

**Verdict at session close**: Ship it. The table renders merchant name + "Visa spend · MCC 5812" secondary + green ✓ when `source_verified=true`. One code file, zero migrations.

## DJ 4 — Sprint 6.5 ops alerts: dedicated table vs execution_log rows

**Question**: Where to persist "we already alerted on this stuck tx" so the 10-min cron doesn't spam the same thing every run?

**Options considered**:
- (A) New `ops_alerts_log` table with (tx_id, alerted_at) — schema migration, single-purpose.
- (B) Reuse `execution_log` with `(entity_type='ops_alert', action='stuck_pending', entity_id=tx_id)`. LEFT JOIN on scan so we filter out already-alerted in one query.

**Chose**: (B) — execution_log.

**Why**: `execution_log` is the existing audit table for "something happened." A dedup row IS something that happened. New table = migration = deploy friction = another schema to maintain. The LEFT JOIN pattern already works for monitor-skips, swap-attempts, and approval-pipeline reviews — we know the indexing story.

**Tradeoff**: `execution_log` gets chattier. Over 6 months with ~10 stuck txs/day that's ~1800 rows — negligible next to the growth_agent volume. If it becomes a problem we add a partial index on `(entity_type, action, entity_id)`.

**Verdict at session close**: Works. Zero-migration deploy. Dedup is checked in the same SQL as the pending-scan — no N+1.

## DJ 5 — Deposit funnel avg_confirm_seconds: drop the metric vs add `confirmed_at` column now

**Question**: The funnel query wrote `AVG(updated_at - created_at) FILTER (WHERE status='confirmed')` but the transactions table has no `updated_at` or `confirmed_at` column. Options: migrate now, or defer.

**Options considered**:
- (A) Ship the migration that adds `confirmed_at timestamptz` + backfill + update all `UPDATE transactions SET status='confirmed'` call sites to also set `confirmed_at = now()`.
- (B) Drop `avg_confirm_seconds` from the response, mark as tech-debt. Shipping one number empty is better than shipping it wrong.

**Chose**: (B) — drop it for now, return `null`.

**Why**: The migration is *not* trivial. Three confirm-write sites (`monitor.ts:215`, `monitor.ts:660`, graceful-shutdown's `failed_restart`), plus backfill for ~2000 existing confirmed rows. If we ship a backfilled `confirmed_at = created_at` we'd get avg confirm = 0 which is actively misleading. Shipping `null` and a "tech-debt" comment is honest and unblocks the rest of Sprint 6.5.

**Tradeoff**: The admin panel shows `—` where a number would be informative. Acceptable for Session 26 ship; logged in the code as the reason.

**Verdict at session close**: Correct call. Chain-health endpoint already gives us the "last confirm age" signal at the chain level, which is the operationally-useful thing. Per-tx confirm latency can wait.

## DJ 6 — Market watcher: generate post copy in-skill vs route through content.ts

**Question**: The new market-watcher cron needs to compose Moltbook post text for new markets. Where should the composing logic live?

**Options considered**:
- (A) Call into `content.ts` (the existing LLM-powered content generator) for every new market. Pass in the question + pools + resolution date, get back a post.
- (B) Template the post locally in `market-watcher.ts` — hardcoded structure with variables filled in.

**Chose**: (B) — local templates.

**Why**: LLM content generation costs tokens, has latency, and produces variable quality. For a utility post like "NEW MARKET: Will X happen? 65% YES — bet here →" we *want* a predictable structure. The template is legible, debuggable, and the preview surfaced to Richard via Telegram is deterministic — he knows exactly what will post. When we want LLM-authored takes (contrarian analysis, market-context commentary), we route through `thought-engine.ts`, not this cron.

**Tradeoff**: Templates feel generic. The same template for a Trump-election market and a BTC-price market reads the same. Acceptable because:
  1. Category emoji differentiates (🗳️ vs 💰 vs 🎭)
  2. Richard sees the preview and can reject/edit before posting
  3. The "smart" layer is the filtering (which markets get proposed at all) — we already skip sports/general by design

**Verdict at session close**: First cycle ran, no false positives. The rate-limit (3 proposals per scan per type) also protects against Richard being flooded when the feed spikes.

## Summary

Six decisions, all pointing toward the same pattern: **exploit existing surface before adding new surface.**
- Multi-chain sell reused wagmi hooks (DJ 1)
- Whitelist reused supported-tokens API (DJ 2)
- Visa polish reused `transactions` row (DJ 3)
- Ops-alert dedup reused `execution_log` (DJ 4)
- Confirmed-at deferred rather than migrated (DJ 5)
- Market watcher reused approval-pipeline instead of rebuilding Telegram round-trip (DJ 6)

Session 26's velocity came from this. When the scaffolding is already there, new features are 50-line adds.
