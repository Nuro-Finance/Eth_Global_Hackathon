## DJ 1 — Chain logos: hand-write SVGs → package (`@web3icons/react`)

**Question**: Richard flagged Arbitrum logo looking wrong + several others suspect. Two paths:
- (A) Spot-check + rewrite my hand-written SVGs against brand guides
- (B) Replace the whole system with a package that has 400+ brand-accurate icons

**Chose**: B.

**Why**: (A) would fix Arbitrum but leave us hand-maintaining 23 SVGs — every chain addition = another hand-drawn SVG. (B) installs `@web3icons/react` (99MB unpacked, but tree-shaken via `/dynamic` subpath), covers 100% of our chains, and any new chain we add is zero extra work. Richard's framing: "get an SVG package that HAS ALL OF THE SVGS WE NEED" — exactly right. Also: brand-accuracy was compromised with hand-drawn; the package uses official press-kit SVGs.

**Tradeoff**: React component instead of static `<img src>`. Used a `<ChainIcon>` wrapper so callers don't care about the delta.

**Verdict at session close**: Right call. Reload flow + transactions table both use it now, no regressions, every chain renders correctly.

## DJ 2 — Allowlist architecture: hardcoded constant → DB-backed table

**Question**: Where should `ERC20_ALLOWLIST` live — in code (as I initially shipped in Session 23 early) or in the DB?

**Options considered**:
- (A) Hardcoded constant in `swap.ts` — fastest to ship, but every token addition = commit + redeploy
- (B) DB-backed table + 60s in-memory cache + admin CRUD endpoints — 2hr refactor, but token adds/toggles become row-edits with no redeploy
- (C) Hybrid: DB for runtime, hardcoded for seed data

**Chose**: B (with the hardcoded data migrated into a seed migration — so C-like for bootstrap, but authoritative in DB).

**Why**: Richard explicitly approved the 2hr refactor. The logic was: we're going to manage 20-50 tokens over the next 6 months. Each code-commit cycle is ~10 min. That's 3-8 hours of pure redeploy time for zero value. Moving to DB means admin can kill a rugged token in seconds + add new tokens without pinging Claude Code. Worth the 2hr upfront.

**Implementation note**: kept `ERC20_ALLOWLIST` as a Proxy-backed export so existing callers don't break. Back-compat without forcing an atomic refactor.

**Risk flagged**: DB-backed means if Postgres is down, no ERC-20 swaps work. Mitigated by keeping the previous snapshot in memory (last-known-good) — snapshot refresh failure doesn't clear the list.

## DJ 3 — Risk architecture: credit card at deposit price vs. post-swap price?

**Question**: Richard pressed on this explicitly. When a user deposits $18 of ANDY, do we credit the card with $18 or with whatever USDC the 0x swap actually returns?

**Options**:
- (A) Credit $18 (deposit-time valuation) — better UX for the user, but we eat the volatility between deposit + swap
- (B) Credit post-swap USDC (our actual output) — user eats volatility, we're a transparent venue

**Chose**: B — and it turned out this is already how the code works. We don't compute "deposit price in USD" anywhere; we compute "USDC received from swap" and that's what bridges to the card.

**Why**: (A) would mean if a memecoin rug-pulls 50% between deposit and our swap, we're eating 50% of the deposit on behalf of the user. At scale that's insurance liability. (B) makes us a payment venue, not a market-maker. Exact same risk model as Coinbase's crypto card.

**Documented**: `Neural Net/Claude Memory/Swap Risk Policy.md` codifies this explicitly with edge-case table (rug mid-transit, pump mid-transit, bank-run on the liquidity pool, 0x route disappearance).

**FE disclosure**: the estimate on the reload screen has `≈` prefix, worst-case slippage line, and a "card credited with actual USDC received" note. Crystal clear to the user.

## DJ 4 — Dynamic chain filter: per-token allowed-chains set

**Question**: Richard caught that chain picker showed all 23 chains even when token was PEPE (Ethereum-only). How do we filter?

**Options**:
- (A) Hardcode per-token chain maps in FE
- (B) Backend `/supported-tokens` already returns each token with its chainId — derive the allowed set on FE from that response
- (C) New backend endpoint `/allowed-chains?token=X` returning just the set

**Chose**: B.

**Why**: `/supported-tokens` already returns the full picture — natives + bluechips + memecoins each tagged with chainName/chainId. All I needed was a 3-line filter on FE. (A) couples FE to backend state (bad). (C) adds a round-trip when we already have the data. (B) is free.

**Implementation**: `allowedChainsForToken` derived via `useMemo` from `supportedTokens` + `selectedToken`. Stable symbols return `null` (null = "all chains OK"), non-stables return the `Set<string>` of allowed chain names. Applied to both reload modals.

**Safety bonus**: auto-bounces `selectedChain` to first allowed when user picks incompatible token. No "Quote unavailable" dead ends possible.

## DJ 5 — Telegram approval UX: polling vs webhook

**Question**: Richard flagged approve/deny buttons as dead. Two fixes:
- (A) Wire up POST /telegram/webhook via nginx reverse proxy — real-time
- (B) Keep polling but drop the interval from 60min to 10s

**Chose**: B for today. A is flagged for Session 24.

**Why**: (A) is architecturally correct — zero polling, zero latency — but requires nginx config changes on VPS + webhook registration with Telegram. That's a ~30 min task with a moving piece (nginx config file on a prod box). Not a casual session-close task. (B) is a 5-line code change that gets tap-to-publish latency from 60+ minutes to 10-20 seconds — which is the UX goal. Meanwhile the webhook wire-up can happen in Session 24 alongside the `/admin` nginx rule that's been pending since Session 22 — batch nginx work.

**Downside of B**: polls Telegram getUpdates every 10s forever. Trivial bandwidth cost, but it's an unbounded background task. Webhook would be more elegant.

**Decision re-visit trigger**: if polling costs show up in logs or we hit rate limits, elevate to the nginx fix immediately.

## DJ 6 — gstack + context7 adoption

**Question**: Richard asked whether to adopt gstack (Garry Tan's YC Next.js starter) + context7 (MCP for live library docs).

**Verdict**:
- **gstack**: NOT wholesale. We're already 80% on that stack (Next.js 15, React 19, next-auth 5, Tailwind 4, Stripe). The differentiators are Drizzle ORM (worth adopting separately as "tech debt retirement" sprint post-funding, 2-3 days) and Upstash Redis (not needed yet at single-node scale). Cloning gstack would rewrite working code for marginal value. Cherry-picking is the right move.
- **context7**: Strong yes. Free. Low risk. Solves a concrete pain point I hit today — wasted 15 min on `@web3icons/react` subpath mismatch that live docs would've caught instantly. Added to `~/.claude.json` alongside our 3 existing MCPs (postgres-cashly, filesystem, github).

**Why the asymmetry**: gstack is an opinionated stack choice that would rewrite working code. context7 is a lookup tool that helps me write better code. Different categories of adoption decision.

## Meta-observation: session rhythm

19 commits in ~7 hours with one mid-session context reset. No rollbacks, no CI failures, no production regressions. Pattern that worked:

- **Ship in small atomic commits** — each commit reviewable on its own
- **Test gates blocked 0 commits today** (verify-deps, typecheck, vitest) — caught nothing because I wasn't sloppy, but they give confidence to move fast

The one stumble: the `@web3icons/react` import path. Should've fetched docs before writing the first import. Context7 fixes this class of error going forward.
