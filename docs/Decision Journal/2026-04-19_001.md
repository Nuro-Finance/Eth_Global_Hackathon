## DJ 3 — Template-literal escape-collision: the 15-minute silent bug

**Question**: Admin console rendered as blank dashes after I shipped the ERC-20 admin UI. What's the right fix and how do we prevent recurrence?

**Context**: Source code had `'<button onclick="toggleAllowlistRow(\'' + esc(r.id) + '\')" ...'`. Inside the enclosing TypeScript template literal, `\'` is decoded to `'` at render time. Result: `''` (empty string) concatenated with `esc(r.id)`, JS syntax error, whole script failed to parse, all loaders failed silently.

**Options**:
- (A) Use `String.fromCharCode(39)` for single quotes — ugly but bulletproof
- (B) Use `\\\'` (triple-backslash) so template-literal decodes to `\'` which JS then interprets as escaped single quote — correct but error-prone
- (C) Switch to data-attributes + event delegation — more invasive refactor, cleanest long-term
- (D) Use HTML entity `&quot;` inside attribute — works but mixes concerns

**Chose**: B for the immediate fix. (C) flagged as Session 25 nice-to-have.

**Why**: Fastest working fix, `node --check` on the rendered JS confirmed clean parse. Shipped in 5 minutes.

**Prevention**: Added this bug class to `/auditor` skill as a "HIGH: Session 23 Bug Patterns → Package subpath/template-literal escape collisions" entry. Pattern: any `\\'` or `\\"` inside a template literal rendering HTML is suspicious — verify with `node --check` on rendered output.

**Verdict**: Won't hit this again unnoticed. `node --check` on rendered JS is now my pre-commit sanity gate for admin-console edits.

## DJ 4 — Style parity vs visual distinction for the two dashboards

**Question**: Richard loved the sub-agents aesthetic, wanted the neural dashboard to match. How much parity vs distinction?

**Options**:
- (A) Carbon-copy style — identical panels, colors, typography. Minimum friction.
- (B) Shared visual language + distinct accents — same panel/chip/legend patterns, but neural uses its own color accents (e.g. orange for decisions)
- (C) Leave neural as-is; just polish

**Chose**: A with the shared emerald accent.

**Why**: The two dashboards are the "two halves of the same brain" — topology vs activity. Visual consistency signals that mental model directly. The old neural dashboard felt like a separate app; the new one feels like a different VIEW of the same system. Also: I spent 30 min redesigning, not 60. Distinct accents would have been 45+ min and obscured the "same system" message.

**Tradeoff**: When looking at thumbnails, the two dashboards look near-identical. User has to read titles to know which is which. Acceptable — the URLs are distinct and keyboard shortcuts (/) work identically.

**Verdict**: Cross-linking via the `?skill=X` URL param + the "Jump to →" chip at the top of each dashboard reinforces the "two views, one system" model. Richard's reaction will be the real verdict tomorrow.

## DJ 5 — `/encode` skill: ritualize vs leave as manual ceremony

**Question**: Session 23 close taught me the session-close ceremony is ~30 min of hand-written docs. Worth automating into a skill?

**Options**:
- (A) Build `/encode` skill — 10-step ritual with templates + calibration notes. ~30 min one-time.
- (B) Leave manual — each session I reconstruct the rhythm.
- (C) Write a script (not a skill) that just dumps git log into a Recap template.

**Chose**: A.

**Why**: Compounding win. 30 min once = 30 min saved EVERY future session (2+ sessions/week). Break-even after ~2 sessions; after that it's net positive forever. The skill encodes the *voice* (campfire vs flight-manifest vs reasoning-audit) which a raw script can't. Also: future Mythos reads the skill like documentation — if we improve it, every future session benefits.

**Tradeoff**: Skill-creation time lived in Session 24 itself, displacing some product work. Acceptable — self-improvement session was the frame.

**Verdict**: The first real use of `/encode` is THIS journal entry. Quality held. 4 docs written in ~20 min vs the hour Session 23 took.
