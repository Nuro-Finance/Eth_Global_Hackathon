# Decision Journal — Session 25 (2026-04-19, entry 002)

Second DJ of the day. First entry (`2026-04-19_001.md`) covered the morning work through the architecture rework. This entry covers the afternoon: the big wallet data-layer decisions, firm Swap execution choices, and a handful of UX calls.

## DJ 1 — Alchemy proxy vs. direct client-side Alchemy calls

**Question**: Should the FE call Alchemy directly or should we proxy through the backend?

**Options considered**:
- (A) Direct client — `NEXT_PUBLIC_ALCHEMY_KEY` baked into the bundle, browser hits Alchemy endpoints directly. Faster first-paint, zero backend involvement.
- (B) Server proxy — FE calls `/api/wallet-portfolio`, which calls `/wallet-portfolio` on the Express backend, which calls Alchemy. Slower, one more hop, but the key stays server-side.

**Chose**: (B) Server proxy.

**Why**: Three reasons.

First, Alchemy API keys are priced per-CU (compute unit). Exposing the key client-side means anyone can sniff it from the network tab and burn our CU budget — classic leak-by-default. Second, the backend proxy lets us cache (30s in-memory per address) which amortizes cost across all users viewing the same wallet. Third, we can rate-limit per-IP cheaply on the backend; client-side has no equivalent.

The extra hop cost is <50ms on the same-host VPS — not meaningful for a dashboard that refreshes every 60s.

**Tradeoff**: Adds a backend endpoint to maintain. Slightly more complex to debug (two layers). Does not scale to a true SPA where backend isn't co-located with FE.

**Verdict at session close**: Correct call. We hit CoinGecko 429s within the first hour of deploy; stale-cache fallback + 5-min TTL kicked in precisely because everything flowed through the server. If we'd gone client-direct, every refresh would have been a cache miss.

## DJ 2 — Auto-extract Alchemy key from RPC_URL_POLYGON instead of adding a new env var

**Question**: The backend needs an Alchemy API key for the new enhanced APIs. Do we add `ALCHEMY_API_KEY` as a new env var, or extract it from an existing var?

**Options considered**:
- (A) Add `ALCHEMY_API_KEY=<KEY>` to `.env`, document it, ask Richard to set it.
- (B) Parse the key out of `RPC_URL_POLYGON` on boot (existing env has `polygon-mainnet.g.alchemy.com/v2/<KEY>` format).

**Chose**: (B) auto-extract on boot.

**Why**: Zero operational cost. The key is already on the VPS (it's been powering the Polygon RPC for months via the same Alchemy account). Alchemy's same API key works across all their supported chains (Ethereum, Base, Arbitrum, Polygon, Optimism, etc.). Adding a new env var means: (a) docs update, (b) another thing Richard has to remember in deploy runbooks, (c) drift risk if the key rotates. None of that adds value.

**Tradeoff**: If Richard ever rotates only the Polygon RPC but not the Alchemy account, we'd silently lose the key. Mitigated by the regex-parser falling through to all `RPC_URL_*` candidates (`RPC_URL_ETHEREUM`, `RPC_URL_BASE`, etc.), so any one of them carrying the key is enough.

**Verdict at session close**: Worked perfectly. `wallet-portfolio-routes.ts` boots with a log warning if no Alchemy key can be found in any RPC URL — good defensive default. No new env var needed, deploy was a one-line pull + pm2 restart.

## DJ 3 — Firm Swap: build new `/quote/swap/firm` endpoint vs. extend existing `/quote/swap`

**Question**: Phase 4 needed a way to get the executable 0x transaction payload (`to`, `data`, `value`, `gas`) for the user's wallet to sign. The existing `/quote/swap` endpoint only returned a preview (buyAmountUsd, minBuyAmountUsd). Extend it, or build a new endpoint?

**Options considered**:
- (A) Extend `/quote/swap` with an optional `taker` query param — when present, return the full tx payload; when absent, return the preview.
- (B) Build a new `/quote/swap/firm` endpoint alongside the existing one.

**Chose**: (B) separate endpoint.

**Why**: Different contracts, different threat models. Preview quotes are rate-limited generously (they're idempotent read-only), firm quotes are burning fresh CU on 0x for each call and should be rate-limited more aggressively. Preview responses can be cached for 30s; firm responses absolutely cannot (the tx data becomes stale the moment block gas prices shift). Mixing them in one endpoint with a conditional branch would either require duplicate caching logic or force both under the stricter policy, wasting preview traffic.

Also: the FE callsites are different. Preview is called on every keystroke (debounced); firm is called once on button-press. Keeping them as separate endpoints means their callers can't accidentally slip into the wrong path.

**Tradeoff**: Some code duplication in the route handler (validation, sellToken resolution, etc.). We mitigated by having both handlers call the same underlying `getNativeSwapQuote` / `getErc20SwapQuote` helpers from `swap.ts`.

**Verdict at session close**: Right call. First test on Base returned a clean tx payload. If we'd overloaded `/quote/swap` with an optional `taker`, we'd still be debugging the cache-bypass conditional.

## DJ 4 — Hide Scams default ON vs. default OFF

**Question**: Once the real All Assets table was wired, Vitalik's wallet showed 25 detected tokens, most of which were obvious airdrop scams ("U S D C ⭐ Visit to claim"). Should we hide scams by default or show everything and trust the user to filter?

**Options considered**:
- (A) Default OFF — show everything, user toggles Hide Scams when they want to filter.
- (B) Default ON — hide scams, show a subtle toggle + count for users who want to inspect them.

**Chose**: (B) default ON.

**Why**: The 95% case on Ethereum EOAs today is "my legitimate holdings are a handful of tokens; the rest is spam airdropped by scammers trying to get me to visit a phishing site". Showing everything by default buries the legit holdings in noise and creates a bad first impression of the app ("why is my wallet full of random shitcoins?"). The 5% case — a power user wanting to inspect airdrop dust — is still one click away.

We also strengthened the heuristic (`looksLikeScam`): beyond the existing zero-price flag, it now matches phishing-style names via regex (`t.me/`, `visit to claim`, emoji clusters, embedded TLDs). Richard's specific example — `{USDC ⭐Visit to claim - | t.me/s...}` — is caught by multiple rules.

**Tradeoff**: A legit brand-new token with no CoinGecko price yet will be hidden as "scam" until its price resolves. Acceptable tradeoff for today — user can toggle off to inspect.

**Verdict at session close**: Richard's verification: "that spam filter works already hahaha". Confirmed. Extended the same filter to Recent Activity later in the day (same reasoning: spam airdrops generate spam transfer entries).

## DJ 5 — Tabs: honest-disabled state vs. clickable-with-Coming-Soon-banner

**Question**: The Swap panel has 4 tabs (Swap, Limit, Buy, Sell). Only Swap is wired. How do we surface the other 3?

**Options considered**:
- (A) Tabs are clickable, clicking a non-Swap tab shows a "Coming soon in Phase N" banner below.
- (B) Tabs are visibly disabled (text-white/25, cursor-not-allowed), hovering shows a tooltip explaining.
- (C) Hide non-functional tabs entirely.

**Chose**: (B) visibly disabled, after flirting with (A) for a day.

**Why**: Richard's feedback on (A) was blunt: "Limit, buy, sell still fake". The dots + clickability of (A) suggested something functional was there. The banner was informational but landed as "oh great, another thing that's not done". The vibe was "things pretending to be ready" — bad signal.

(B) inverts that: the tab LOOKS dead, hover tooltip explains why. User immediately sees "these aren't for me right now" and moves on. No cognitive load, no disappointment.

(C) was tempting for pure honesty but loses context about the product roadmap — a user seeing "Swap" alone wouldn't know Limit/Buy/Sell are planned. (B) keeps the scaffold visible while being honest about state.

**Tradeoff**: Slightly less dramatic "reveal" when Limit or Buy eventually ship — the tab goes from greyed-out to active rather than appearing new. Fine; less disorienting for existing users.

**Verdict at session close**: Still waiting on Richard's feedback post-deploy. But based on his earlier reaction to (A), (B) should land well. Kept the markup simple: no banner, no "coming soon" chip, no dots — just a muted tab with a hover hint.
