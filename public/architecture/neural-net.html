08 · Mythos brain

Neural Net — how Mythos thinks

Four cross-linked dashboards visualize different slices of the Mythos orchestration architecture. A user query flows through the orchestrator, which dispatches to skills (on-demand, scheduled, kernel), which read + write artifacts (Neural Net docs, source code, VPS state). Every skill invocation logs to execution_log, feeding the live /api/skill-health endpoint that the dashboards consume. Scroll through the story below, then explore the live dashboards.

01 — The story (scroll →)

Five frames, left-to-right. How a single query turns into a decision, then gets recorded for the next session.

01

A query arrives

Richard types a message, or a cron wakes up. Either way a new intention enters the system. Mythos doesn't answer directly — it routes through the orchestrator first.

user → mythos
"audit the reload flow"
// intent detected
02

Orchestrator picks a skill

The orchestrator maps the query against the skills/ directory. Keywords + verbs + context. Picks one, or chains two. Never guesses a skill name — only dispatches to names it can see.

mythos → /ux-reviewer
// matched on "review" + "flow"
→ Skill tool call
03

Skill reads artifacts, does work

The skill touches files (Neural Net docs), runs commands (git log, pm2 list), makes API calls (Claude_in_Chrome MCP). Every file read is recorded on the decision trace.

/ux-reviewer reads:
reload-ui.html
deposit-flow.html
→ runs navigate + read_page
04

Result + telemetry

Skill returns its answer to Mythos. The SubagentStop hook fires, SSH's to the VPS, and writes a row to execution_log with entity_type='claude_skill'. Closes the old gap where on-demand skills left no trail.

INSERT INTO execution_log
(entity_type='claude_skill',
  entity_id='ux-reviewer',
  action='run', status='ok')
05

Health dashboards update

Within minutes, /api/skill-health reflects the new activity. Sub-Agents graph shows the node lit up. Neural Map records the decision trace. Unified dashboard overlays both. Next session, Mythos boots with this context.

GET /api/skill-health
→ [{skill:'ux-reviewer',
  ok:1, err:0, last:'2s'}]
// node glows green

04 — How the telemetry works

SubagentStop hook — closing the telemetry gap

Before Session 25 only 7 cron-driven skills had execution_log rows. On-demand skills (auditor, bridge, deployer, encode, boot, gate-check, ux-reviewer) invoked via the Skill tool left no DB trail. The SubagentStop hook in ~/.claude/settings.json now calls ~/.claude/hooks/log-skill-invocation.js, which SSH's to the VPS and writes entity_type='claude_skill' rows. Health + invocation-heat dashboards pick them up within minutes.

What goes into execution_log

  • entity_type — always claude_skill for skill invocations; other values for cron tasks, webhooks, swap executions
  • entity_id — the skill name (ux-reviewer, auditor, etc.)
  • action — run for a full invocation, finer-grained for sub-steps
  • status — ok, error, or skipped
  • metadata — free-form JSON (elapsed_ms, error_msg, correlation_id if chained)

Health classification (what the rings mean)

  • green — no errors in 24h
  • yellow — 1–3 errors in 24h OR <30% error rate
  • red — >30% error rate OR 5+ consecutive failures
  • gray — no activity in 24h (idle, not broken)
Why this matters. Before Session 25 there was no way to answer "is skill X healthy?" without grepping logs. Now the dashboards give you a live answer, the hook keeps the data fresh, and the unified view connects which skill handled which decision-step. The Mythos brain is observable.
