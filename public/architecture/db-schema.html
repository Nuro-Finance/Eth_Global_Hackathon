Cross-domain gotchas (things that burned us)

  • Mixed id types — users.id is VARCHAR, agents.id is UUID, cards.user_id is VARCHAR(36). Cross-type FKs fail at DDL time. Always query information_schema.columns before writing a new FK.
  • CREATE TABLE IF NOT EXISTS silently no-ops on pre-existing tables — if the table exists but columns differ, your migration is a liar. Session 21 migration 010 incident: agents pre-existed with different id type; migration silently did nothing; three hotfix migrations required.
  • String-valued taxonomy columns — no CHECK constraints on status, category, destination, payout_destination. The app layer owns the taxonomy; new values add zero migrations. Parse prefix:args compound strings (agent:<uuid>, community:<id>, external:0x<addr>) in handler code.
  • metadata JSONB on every table where future enrichment is plausible (card_settlements, agent_bets, user preferences). Avoids the "ALTER TABLE ADD COLUMN" migration tax.
