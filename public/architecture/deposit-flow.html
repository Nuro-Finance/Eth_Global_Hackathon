02 · Critical path

Deposit → Card credit

The canonical money-movement path, end-to-end. User sends tokens to their HD-derived deposit address on any supported chain; Monitor detects the balance, Swap converts to USDC (if not already USDC), Bridge routes to Base, and SD3 Issuer credits the Visa card. Every other deposit path is a variant of this one. Below: the full sequence, then a focused view of the bridge tier (CCTP-direct vs LZ→CCTP 2-hop), then a worked example with concrete addresses and transaction shapes.

03 — Worked example

Concrete walk-through of what a single reload looks like. Addresses are illustrative placeholders; real txs are visible in execution_log by correlation_id.

Scenario — User 42 reloads with 50 SHIB from Ethereum
1

User sends SHIB to deposit address

Frontend derives address via /api/supported-tokens. User hits "Send from Wallet" → wagmi calls SHIB.transfer(0xDEP…42A1, 50 * 10^18).

source chain: Ethereum (chainId 1) · tx lands ~12s later
2

Monitor poll cycle detects balance

Every POLL_INTERVAL_MS (60s when live), Monitor scans the allowlist × user-deposit matrix. SHIB balance > 0 triggers swapInflight.set(...) to prevent double-swap, then calls executeErc20ToUsdcSwap.

table rows touched: erc20_allowlist, users, user_deposits · writes to execution_log (action=monitor_tick)
3

Approve + swap via 0x

Backend signer (the Deployer key at 0x27FbEAD2…ECBC) signs two txs:

SHIB.approve(0xAllowanceHolder, 50e18) 0xAllowanceHolder.swap(SHIB → USDC, recipient=0xDEP…42A1)

Quote is re-fetched at execution time from 0x /swap/allowance-holder/quote — the FE preview quote is never trusted.

~$0.10 USDC lands at same deposit address
4

Next poll detects USDC → Bridge decision

Ethereum is CCTP-enabled (domain 0) → skip the LayerZero hop. bridge.ts::bridgeToBase builds:

USDC.approve(tokenMessenger, amount) tokenMessenger.depositForBurn(amount, destDomain=6, mintRecipient=userIssuerAddr)
if user were on HyperEVM instead, step 4a would be OFTadapter.send(...) to Arbitrum hub first
5

Circle attestation (async wait)

Typical wait: 10–15 minutes. Monitor's cctp_finalize reconciler polls Circle's attestation API; once signed, it calls messageTransmitter.receiveMessage(msg, attestation) on Base to complete the mint.

alert threshold: > 30 min attestation delay → Telegram ping to admin
6

SD3 webhook → card credit

USDC lands at the Issuer-linked Base address that SD3 provisioned during KYC. SD3 fires an inbound-deposit webhook to our backend, we post the entry into transactions, cards.balance increments, FE refreshes via next SWR fetch.

end-to-end: ~1–3 min (Base-direct USDC) · ~15–25 min (native swap path) · ~10–15 min (Solana)

Four variants to know

  • Native path (Marathon 7): Monitor detects ETH/MATIC/BNB/AVAX/S/HYPE → swap.ts::executeNativeToUsdcSwap routes through 0x. USDC lands at the same address; next poll cycle picks it up and takes the bridge path below.
  • ERC-20 path: Monitor detects an allowlisted token (SHIB/PEPE/PENGU/ANDY/LINK/UNI/WBTC/WETH/cbBTC) → executeErc20ToUsdcSwap runs approve + swap. Same terminus.
  • Base-direct: USDC already sitting on Base → skip CCTP entirely, direct send to the Issuer-linked Base address.
  • Solana: Solana USDC → @circle-fin/bridge-kit (not Circle's public CCTP SDK — that one still lacks remote domains) → Base mint → Issuer.

Why the Monitor polls twice

For swap paths, the same address needs to be polled twice — first to detect the native/ERC-20 balance, then again after the swap completes to see USDC. The swapInflight map prevents double-swaps during the gap. 5-minute TTL.

Risk policy applies here. The card gets credited with actual USDC received, not the pre-swap dollar value of the deposit. If a memecoin rugs 30% between deposit and swap, user eats the loss. See on-chain routing for edge cases and Neural Net/Claude Memory/Swap Risk Policy.md for the full codification.
