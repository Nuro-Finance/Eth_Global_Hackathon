09 · End-to-end

Full request lifecycle — Reload Card

The complete happy-path from "user clicks Reload" to "card balance updated". Nine actors: Browser → Next.js FE → Next.js /api proxy → Express BE → Postgres → 0x API → source chain → Base chain → SD3 Issuer. Synchronous through step 12; asynchronous past that. Below: the sequence, then timing breakdown, then a failure-mode flow so you can see where retries kick in.

04 — Notes & risk disclosure

Failure domains, spelled out

  • Quote phase fails — /quote/swap returns degraded: true. FE shows "quote unavailable"; user can still proceed (execution-time quote will be re-fetched, never trusting the preview).
  • Swap reverts — logged to execution_log with reason code. Native balance stays put. User can retry; swapInflight TTL (5 min) clears automatically.
  • CCTP attestation delay > 30min — admin gets a Telegram alert. Usually Circle infrastructure; manual retry via messageTransmitter.receiveMessage clears it.
  • SD3 webhook doesn't fire — Monitor's issuer_sync reconciler catches it on the next cycle via GET /balances comparison. No silent-loss scenarios.

Every step is recorded

Every on-chain tx carries a correlation_id that ties the quote, swap, bridge, and credit rows together in execution_log. Post-incident debugging is a single query.

Risk disclosure. The card gets credited with the actual USDC that lands, not a pre-swap dollar estimate. If a memecoin rugs 30% mid-transit, the user eats that delta. See diagram 02 + the Swap Risk Policy doc.
