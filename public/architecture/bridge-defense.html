11 · Security

Bridge defense posture

AFI's omnichain bridge runs a defense-in-depth posture against the dominant DVN-compromise attack class — RPC poisoning of the observation layer that lets a forged source-chain message pass cryptographically valid attestation. The protocol ships five on-chain + off-chain controls (per-message cap, per-peer rolling 24h cap, single-block on-chain pause, off-chain reserve reconciliation, TS-layer kill-switch) with a multi-DVN attestation quorum. Every layer is independent — compromise of any single one does not breach the bridge.

Aligns to LayerZero's official integration checklist (April 2026): "The LayerZero Labs DVN will not sign or attest messages from any applications that utilize a 1/1 configuration."

02 — Attack vector (LayerZero's official disclosure)

LayerZero's April 2026 statement on the dominant DVN-compromise class: "compromise[d] a quorum of the RPCs the LayerZero Labs DVN relied upon...swap[ping] out binaries running the op-geth nodes." Attackers then DDoS'd the remaining healthy RPCs to force failover to the poisoned pair. The DVN attestation was cryptographically valid — its view of the source chain was lied to. Not a signature bug, not a replay, not a missing nonce check. The observation layer was corrupted, and a 1-of-1 DVN configuration has no cross-check.

Threat-actor attribution for the exemplar incident (preliminary): DPRK Lazarus subgroup TraderTraitor — same operational cadence as adjacent multi-hundred-million-dollar exploits in the same window. The attack class is now table-stakes for any cross-chain protocol's threat model.

04 — The five controls in detail

1. On-chain per-message cap (100k USDC default). A single forged message can drain at most the cap, not the entire escrow. A nine-figure attack would require thousands of forged messages through this contract — each must pass the DVN quorum, so the attacker's operational cost scales linearly with proceeds. Tunable by owner; default sized at ~10× typical user transfer.

2. On-chain per-peer 24h cap (500k USDC default). Rolling window bounds daily drain from any single compromised pathway. Even if the per-message cap is bypassed via many small messages, the pathway hits the daily ceiling and reverts. State: cumulativeReceivedInWindow[srcEid] + windowStartTimestamp[srcEid] map on the adapter.

3. Single-block owner pause. setPaused(true) halts all inbound messages in one block (~1 second on Arbitrum). Industry incidents have shown multi-step manual pauses taking tens of minutes — during which replay attempts hit the contract and only fail because the hot wallet was drained first. Our pause stops the pathway at the receive handler, not at downstream wallets.

4. Off-chain reserve reconciliation monitor. src/lz-reserve-monitor.ts polls every 5 min: reads Arbitrum OFTAdapter USDC balance, reads totalSupply() of MyOFT on each of 5 spokes (zkSync, Scroll, Celo, Gnosis, BSC), computes drift = hub - sum(spokes). Alert at drift < -$100 USDC. The dominant attack shape is exactly this: spoke-side mint without hub-side escrow. If an attacker mints on zkSync without depositing on Arbitrum, we see the drift within 5 min of the next cycle.

5. TypeScript-layer kill-switch flag. CONFIG.LZ_BRIDGE_ENABLED defaults to false. Backend service refuses to initiate any LZ send via bridge.ts unless the flag is explicitly flipped to true in the VPS .env. Independent of on-chain paused state — belt-and-suspenders. Even if an attacker got the owner key and un-paused the contract, the backend layer would independently refuse.

Primary path is not LayerZero at all. Circle CCTP handles the bulk of deposit volume (Base, Ethereum, Arbitrum, Optimism, Polygon, Avalanche). CCTP messages are signed by Circle's attestation service — cryptographically verified per-message, not DVN-verified. The DVN-compromise attack class does not apply to CCTP. LayerZero is only the first hop from 5 exotic chains Circle doesn't cover.

05 — Multi-DVN quorum (target config)

The hardened config at layerzero.config.hardened.ts defines the 2-of-3 target per pathway: LayerZero Labs DVN + Nethermind DVN + Google Cloud DVN. DVN addresses verified on-chain (eth_getCode) on 2026-04-22, which surfaced a separate finding: a prior config had Arbitrum and zkSync DVN addresses that were NOT deployed contracts (Arbitrum's was on a different chain; zkSync had a typo). Those two pathways had been dead-on-arrival since deployment — coincidentally safer than a working 1-of-1 config, but coincidentally so, not by design. Both replaced.

Migration sequence: (a) compile hardened MyOFTAdapter on Node 22 (Hardhat v3 incompatible with Node 23), (b) deploy new contract on all 6 chains, (c) lz:oapp:wire --dry-run, (d) operator reviews config delta, (e) real wire, (f) flip LZ_BRIDGE_ENABLED=true, (g) scheduler picks up the reserve monitor. Gated behind a Node 22 environment fix on the VPS.

Independence of DVN operators: LayerZero Labs, Nethermind, Google Cloud are operationally independent — different companies, different keys, different infrastructure. An attacker would need to compromise ≥2 simultaneously to forge a quorum, substantially higher cost than the single-DVN-RPC-set compromise that has dominated DVN-class incidents. Independence is reviewed quarterly; if two operators we rely on merge infrastructure or leadership, the quorum weakens and we re-pick.

06 — Helm hardens the bridge: ideal-state flow

The bridge controls above are static — they enforce the same caps regardless of what's calling them. Helm is AFI's behavioral security plane (Marathon 8 Corvus, Layer 4.5): a 52-rule catalog that watches every I/O surface across the agent stack, including the bridge. The diagram below shows the bridge with Helm in ideal state — every applicable rule active and gating its surface.

The diagram is a forward-looking design view: it shows what the architecture looks like with all 52 catalog rules enforced. The TypeScript bridge today enforces a subset (HELM-108 secret redaction live, HELM-101 egress observer in observe mode); the rest scaffold into the Rust ingress proxy in Phase 1 of Marathon 8.

What you're looking at. Every dashed arrow is a Helm gate. The bridge data path (cyan) flows top-to-bottom; Helm (green) watches each transition. A bridge call doesn't reach send() until ingress screening passes. The signed tx doesn't broadcast until egress allowlist + value cap + bridge-adapter allowlist all pass. The credential to sign is issued by Helm's vault with a 5-minute TTL — even if the agent's process is compromised, the bridge-owner token expires before it can be re-used. Every emitted event is scanned for secret leakage before it lands in logs. Compound critical-rule hits (HELM-501) and any bypass reasoning (HELM-502) jump straight to Quarantine, which halts the bridge data path at the source.

Defense-in-depth restated: an attacker who compromises one layer (RPC poisoning, owner key theft, or agent jailbreak) cannot drain the bridge in isolation — the next layer's controls fire independently. Helm is the meta-layer: it makes the per-layer controls visible, audit-loggable, and centrally tunable.

07 — Operational posture going forward
  • No new LayerZero protocol integration without multi-DVN baseline. LZ Labs mandate makes this enforceable at their attestation layer; we also self-enforce at layerzero.config.hardened.ts.
  • DVN operator independence reviewed quarterly. If two DVNs we rely on merge infrastructure or leadership, their independence degrades and the quorum weakens.
  • Reserve drift monitor is always-on. Scheduler cron: */5 * * * *. First line of detection.
  • Owner key on hardware wallet + 2-of-3 multisig. No single signer can unpause. No single signer can raise the per-message cap.
  • Helm posture promotes through observe → enforce as the rule surface widens. HELM-101 egress is in observe mode today; flips to enforce once the full axios-client surface is instrumented (Marathon 8 Phase 1 work). Operator can flip per-rule from the admin tab without redeploy.
  • Public post-mortem on any security event. Industry standard since the 2026 wave of cross-chain incidents — full transparency on root cause, controls that fired, controls that didn't, and forward-looking changes.
