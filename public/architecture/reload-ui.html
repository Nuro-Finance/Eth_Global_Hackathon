02 — Where the deposit address comes from

The deposit address the user sees is not a pre-provisioned pool address. It's derived on the fly from a master seed using BIP-44 HD derivation, scoped to the user's ID. Every user gets a unique address on every supported chain, and the backend can recover the private key at any time because the derivation is deterministic. No key storage, no rotation, no custody risk on deposit keys (spending keys are separate).

Same address across EVM chains. Because all EVM chains share the secp256k1 curve and the same address encoding, User 42's address on Ethereum, Arbitrum, Polygon, Base, Optimism, etc. is the same 20-byte address 0xDEP…42A1. We only need to derive once per user for all 22 EVM chains. Solana uses ed25519 + a different encoding, so it gets its own path and a separate base58-encoded address.
Worked example — how User 42 got their deposit address
1

User signs up, gets assigned userId=42

Persisted to users.id (VARCHAR on prod). This integer becomes the HD derivation index for every subsequent address.

2

Backend derives on first /api/supported-tokens call

Inside deposit.ts::deriveUserDepositAddress:

const hdNode = HDNode.fromMnemonic(MASTER_SEED) const userNode = hdNode.derivePath(`m/44'/60'/0'/0/42`) const evmAddress = userNode.address // 0xDEP...42A1
3

Cached in user_deposits to skip re-derivation

Row: { user_id: '42', chain: 'ethereum', address: '0xDEP...42A1' }. Subsequent requests return from DB instead of re-deriving. The private key is never persisted — it's re-derived from the master seed whenever we need to sign a swap or bridge tx.

4

FE displays 0xDEP…42A1 + QR

The same address is shown regardless of which EVM chain the user picks. Only the chain-picker value changes, which dictates how Monitor routes the deposit downstream. Solana picks a different address derived from the Solana path.

5

Recovery guarantee

If the user_deposits table was nuked, we could re-derive every user's address from MASTER_SEED + userId. That's why the seed is rotated via a BIP-39 split with 3 officers (Richard + 2 backups), not left in a single .env.

KMS rotation is a Sprint 3 priority — currently all signing still happens in-process with the raw seed in memory
03 — Implementation notes

Two parallel reload UIs in the codebase

  • src/features/dashboard/my-card-1/components/ReloadFlow.tsx — the side-panel reload on the My Card page. Wider canvas, chain picker opens a full-height sheet.
  • src/features/dashboard/my-card-v2/ReloadModal.tsx — the popup reload on the Overview page. Modal style with category tabs in the token-pill dropdown.

Both share the same backend endpoints (/api/supported-tokens, /api/quote/swap) and both implement dynamic chain filtering. Both were upgraded to the 3-category picker in Session 23 (commit 96513a1) and cross-filter + style parity in Sessions 23–24.

Dynamic chain filter — the bug-prevention move

Prior to Session 23 the chain picker showed all 23 chains even when the user had picked a token that only existed on Ethereum. That led to silent "Quote unavailable" dead-ends. The fix derives an allowed-chains set from the /api/supported-tokens response per token, then filters the chain grid to that set + auto-bounces selectedChain if the user's current pick becomes invalid.

Risk disclosure inline

The estimate has a ≈ prefix, a worst-case-slippage line, and a below-threshold warning when the quote is under $5 USDC. No surprise for the user — they see the minimum they'll receive before they send.
